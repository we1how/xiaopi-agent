## 用户偏好

### 规划风格
- **详细程度**: 极度具体，每个步骤必须有文件路径和函数名
- **阶段划分**: 大型功能必须分Phase交付
- **风险评估**: 每个步骤必须有风险等级和缓解方案

### 禁止事项
- 模糊描述（"修改相关文件"）
- 缺少测试策略的计划
- 不能独立交付的阶段

---

## 最佳实践（已验证）

### 规划模板
1. 需求分析 → 澄清、成功标准、约束
2. 架构审查 → 受影响组件、可重用模式
3. 步骤分解 → 具体行动、文件路径、依赖、风险
4. 实施排序 → 依赖优先级、上下文最小化
5. 风险评估 → 风险矩阵、缓解方案
6. 测试策略 → 单元/集成/E2E覆盖

### 阶段划分标准
- **Phase 1**: MVP - 最小可用切片
- **Phase 2**: 核心体验 - 完整正常路径
- **Phase 3**: 边缘情况 - 错误处理、边界
- **Phase 4**: 优化 - 性能、监控
